      <h2 id="背景"><a href="#背景" class="headerlink" title="背景"></a>背景</h2><p>最近在做内容审核时，需要把大量对象存储（OBS / COS）上的图片交给多模态大模型走一遍，期望对每一批图片产出统一的判定结果。</p>
<p>原方案是把所有图片 URL 一次性塞进同一个 JSON 请求体，看起来很省事，但只要图片一多，就容易遇到两个问题：</p>
<ol>
<li>请求体里的 Base64 文本非常大，HTTP 客户端序列化时堆内存直接拉满；</li>
<li>部分大模型对单次请求的图片数量、整体大小有限制，超过就 400 或者直接超时。</li>
</ol>
<p>于是就有了这次重构：</p>
<ul>
<li>使用轻量级 <strong>OkHttp 4.x</strong> 作为 HTTP 客户端，单例 <code>OkHttpClient</code> + 连接池；</li>
<li>对 OBS / COS 的签名链接做<strong>流式下载 → 实时 Base64 编码</strong>，避免一次性读入完整 <code>byte[]</code>；</li>
<li>把图片列表按<strong>小批量（batch）分页</strong>循环处理，每批结束释放临时对象；</li>
<li>所有 <code>Closeable</code> 都用 <strong>try-with-resources</strong> 包裹，杜绝连接/文件句柄泄漏。</li>
</ul>

<h3 id="关于内存优化的几点修正"><a href="#关于内存优化的几点修正" class="headerlink" title="关于内存优化的几点修正"></a>关于内存优化的几点修正</h3><p>原贴里有两处容易误导的写法，这里特意拎出来订正：</p>
<ol>
<li><strong><code>base64Data = null</code> 并不能释放内存</strong>。Base64 字符串在循环里已经塞进 JSON，请求体中仍持有引用，把变量置 <code>null</code> 既不会触发 GC，也不会让对象更快回收；真正有用的是把<strong>整个 <code>JSONArray</code> 的生命周期</strong>控制在方法栈内，方法结束请求体就不可达。</li>
<li><strong>不要把 <code>System.gc()</code> 当作调优手段</strong>。它是同步 Full GC，会导致所有工作线程 Stop-The-World；只有在<em>确认</em>接下来要分配超大对象、并且可以接受 STW 暂停时才考虑。</li>
</ol>
<p>实际能落地的内存优化只有三件事：<strong>控制单次请求体积、复用 OkHttpClient 减少对象分配、用 try-with-resources 释放外部资源</strong>。</p>
<h2 id="三、按批次循环调用"><a href="#三、按批次循环调用" class="headerlink" title="三、按批次循环调用"></a>三、按批次循环调用</h2><p>当你需要处理成千上万张图片时，永远不要一次性塞给大模型，按 <code>batchSize</code> 分页，每批结束让局部变量自然失效，下一次 Young GC 就能回收：</p>

<h2 id="四、还可以再做的优化"><a href="#四、还可以再做的优化" class="headerlink" title="四、还可以再做的优化"></a>四、还可以再做的优化</h2><p>在生产环境把上面这套跑稳之后，还可以继续打磨：</p>
<ul>
<li><strong>客户端压缩</strong>：OkHttp 注册 <code>GzipInterceptor</code>，减少下行体积；同步 <code>addInterceptor</code> 给 RequestBody 也压一下，能省 60%+ 带宽。</li>
<li><strong>连接池调优</strong>：监控 <code>ConnectionPool.connectionCount()</code>，根据下游大模型网关的连接上限反推 <code>maxRequestsPerHost</code>。</li>
<li><strong>限流与失败重试</strong>：用 Resilience4j / Guava Retrying 做退避，避免把网关打挂。</li>
<li><strong>请求体不要走 <code>String.getBytes()</code></strong>：使用 <code>RequestBody.create(mediaType, byte[])</code> 或 <code>BufferedSink</code> 直接写流，避免大字符串驻留堆。</li>
<li><strong>关闭大图片</strong>：在 <code>convertUrlToBase64</code> 里限制 <code>Content-Length</code> 或读取总字节数，超过阈值直接拒掉，防止异常图片撑爆内存。</li>
</ul>
<h2 id="五、小结"><a href="#五、小结" class="headerlink" title="五、小结"></a>五、小结</h2><p>这次重构的核心其实只有一句话：<strong>把“一次大请求”拆成“多次小请求”，让堆里的临时对象随方法栈出栈而自然释放</strong>。OkHttp 的连接池、try-with-resources、分批循环，是 Java 后端做高吞吐图像 AI 推理时最实用的三件套。</p>
