Java 批量流式调用大模型：OkHttp 封装与内存释放实战

2026-07-14
未分类
7.1k 词

背景

最近在做内容审核时，需要把大量对象存储（OBS / COS）上的图片交给多模态大模型走一遍，期望对每一批图片产出统一的判定结果。

原方案是把所有图片 URL 一次性塞进同一个 JSON 请求体，看起来很省事，但只要图片一多，就容易遇到两个问题：

  1. 请求体里的 Base64 文本非常大，HTTP 客户端序列化时堆内存直接拉满；
  2. 部分大模型对单次请求的图片数量、整体大小有限制，超过就 400 或者直接超时。

于是就有了这次重构：

  • 使用轻量级 OkHttp 4.x 作为 HTTP 客户端，单例 OkHttpClient + 连接池；
  • 对 OBS / COS 的签名链接做流式下载 → 实时 Base64 编码，避免一次性读入完整 byte[]；
  • 把图片列表按小批量（batch）分页循环处理，每批结束释放临时对象；
  • 所有 Closeable 都用 try-with-resources 包裹，杜绝连接/文件句柄泄漏。

关于内存优化的几点修正

原贴里有两处容易误导的写法，这里特意拎出来订正：

  1. base64Data = null 并不能释放内存。Base64 字符串在循环里已经塞进 JSON，请求体中仍持有引用，把变量置 null 既不会触发 GC，也不会让对象更快回收；真正有用的是把整个 JSONArray 的生命周期控制在方法栈内，方法结束请求体就不可达。
  2. 不要把 System.gc() 当作调优手段。它是同步 Full GC，会导致所有工作线程 Stop-The-World；只有在确认接下来要分配超大对象、并且可以接受 STW 暂停时才考虑。

实际能落地的内存优化只有三件事：控制单次请求体积、复用 OkHttpClient 减少对象分配、用 try-with-resources 释放外部资源。

四、还可以再做的优化

在生产环境把上面这套跑稳之后，还可以继续打磨：

  • 客户端压缩：OkHttp 注册 GzipInterceptor，减少下行体积；同步 addInterceptor 给 RequestBody 也压一下，能省 60%+ 带宽。
  • 连接池调优：监控 ConnectionPool.connectionCount()，根据下游大模型网关的连接上限反推 maxRequestsPerHost。
  • 限流与失败重试：用 Resilience4j / Guava Retrying 做退避，避免把网关打挂。
  • 请求体不要走 String.getBytes()：使用 RequestBody.create(mediaType, byte[]) 或 BufferedSink 直接写流，避免大字符串驻留堆。
  • 关闭大图片：在 convertUrlToBase64 里限制 Content-Length 或读取总字节数，超过阈值直接拒掉，防止异常图片撑爆内存。

五、小结

这次重构的核心其实只有一句话：把“一次大请求”拆成“多次小请求”，让堆里的临时对象随方法栈出栈而自然释放。OkHttp 的连接池、try-with-resources、分批循环，是 Java 后端做高吞吐图像 AI 推理时最实用的三件套。
